# Performance

This page is the engineering-credibility document. The choices below are the difference between a chart that ticks at 60 Hz and one that drops a frame every time a sample arrives.

## Why Canvas, not SVG

An SVG chart driven through React re-renders the chart's subtree on every value update, so each frame pays for React reconciliation **plus** the SVG/DOM mutate + layout + paint. Both costs grow with the number of on-screen elements, and past a few hundred they blow the 16 ms budget — visible jank, dropped frames, and layout thrash on adjacent UI.

This is measured, not assumed. The benchmark in `scripts/bench/` (`apps/demo/bench/`) renders N telemetry widgets three ways — React-driven SVG, imperative SVG (one shared rAF, mutate by ref, no React), and Canvas — and on a desktop i9 / Chromium at 60 fps:

- **React-driven SVG** starts dropping frames around **N ≈ 300** widgets, and by **N ≈ 50** once the CPU is throttled to mid-tier-laptop speed.
- **Imperative SVG** roughly doubles that headroom (drops ≈ N ≈ 600) — so about half of React-driven SVG's per-frame cost is React reconciliation, not SVG.
- **Canvas** keeps per-frame main-thread work ~10× lower than imperative SVG (≈ 1 ms vs ≈ 5 ms at N = 200) and holds 60 fps furthest, including N = 200 on throttled hardware where both SVG paths fail.

So Canvas is the default for the **high-frequency** charts — `TimeSeries`, `MultiAxisPlot`, `Attitude` — which write directly to a Canvas via `requestAnimationFrame`:

```
sample arrives  →  push into RingBuffer (no setState)  →  next rAF tick reads buffer + paints
```

Lower-frequency widgets don't need it: `Gauge` is SVG (a single needle updated via React state is cheap), and the numeric `SignalPanel` is plain DOM. Match the tool to the update frequency. React state is reserved for UI concerns — connection status, channel config, themes — and stays out of the chart hot paths.

## RingBuffer — bounded memory, fast reads

Plain JS arrays grow unbounded. Push `[]`-into-`.push(x)` at 1 kHz and within minutes you've allocated tens of megabytes of garbage.

Altara stores telemetry in a `RingBuffer`:

- Backed by `Float64Array` (typed array, no boxing, contiguous memory).
- Fixed capacity — overwrites oldest sample once full.
- `getValues()` / `getTimes()` return ordered slices in O(capacity).

### Reading without allocating

`getValues()` / `getTimes()` each **allocate a fresh `Float64Array`** per call — convenient, but not free in a 60 fps loop. Keeping React out of the hot path (no `setState` per sample) removes reconciliation, but it does **not** by itself make the loop allocation-free: a chart that calls `getValues()`/`getTimes()` for an extent pass and again for a draw pass churns ~4 arrays per channel per frame.

How much that matters is load-dependent, and the benchmark measures it (`MODE=alloc` in `scripts/bench/`). On a desktop i9, even 96 such allocations per frame (24 channels × full 10k buffers) never drops a frame — V8's generational GC absorbs the short-lived arrays — but it does produce GC pauses up to ~15 ms that eat into the frame's headroom, and once the CPU is throttled to mid-tier speed those pauses turn into **dropped frames** (hundreds over 30 s) that the allocation-free path avoids entirely.

So for the hot path, read into a reused buffer with `readInto()` / `readTimesInto()`, which fill a caller-owned `Float64Array` and allocate nothing — this is what `TimeSeries` and `MultiAxisPlot` do internally:

```tsx
import { useTelemetry } from '@altara/core';

function CustomChart({ source }) {
  const { buffer } = useTelemetry(source, 10_000);
  const scratch = useRef(new Float64Array(10_000)).current;
  // inside your rAF loop — zero allocation per frame:
  //   const n = buffer.readInto(scratch);
  //   for (let i = 0; i < n; i++) { /* read scratch[i] */ }
}
```

`getValues()` / `getTimes()` stay as the convenient choice for one-off, non-hot-path reads.

## `windowMs` vs `bufferSize`

Two knobs that look related but do different jobs:

- **`windowMs`** — the visible time window. Only samples newer than `now - windowMs` are rendered.
- **`bufferSize`** — the per-channel ring-buffer capacity (samples). Older samples beyond this drop.

Sensible defaults by scenario:

| Scenario | `fps` | rate | `windowMs` | `bufferSize` |
|---|---|---|---|---|
| IMU summary view | 60 | 100 Hz | 30_000 (30 s) | 5_000 |
| Battery / temp dashboard | 30 | 1 Hz | 600_000 (10 min) | 1_000 |
| 1 kHz vibration analyzer | 60 | 1000 Hz | 5_000 (5 s) | 10_000 |
| 24-hour facility monitor | 15 | 0.1 Hz | 86_400_000 | 50_000 |

Pick `bufferSize ≥ rate × windowMs / 1000` so you have at least the visible window's worth of data.

## High-frequency feeds: the Web Worker pipeline

Above ~500 Hz, even decoding the WebSocket frames on the main thread starts to nibble at the frame budget. `createWorkerDataSource` moves the entire ingest path off-thread:

```ts
import { createWorkerDataSource } from '@altara/core';

const source = createWorkerDataSource({
  url: 'ws://localhost:9090',
  subscribeMessage: { op: 'subscribe', topic: '/imu', type: 'sensor_msgs/Imu' },
  flushHz: 60,
  // Functions can't be cloned across worker boundaries, so the extractor
  // is a source string evaluated inside the worker.
  extractorSource: 'function (msg) { return msg.angular_velocity.z; }',
});
```

The worker:

1. Owns the WebSocket. Decoding, parsing, and the `valueExtractor` all run off the main thread.
2. Batches incoming samples in memory.
3. Posts batches to the main thread at `flushHz` (default 60), independent of the inbound rate.

Net effect: the main thread only sees ~60 messages per second regardless of whether the source is publishing at 100 Hz or 10 kHz.

## FPS tuning

`fps` defaults to 60. Drop it when:

- You're rendering **many** charts at once (4+) — each `TimeSeries` runs its own rAF loop.
- You're targeting battery-constrained mobile.
- Your data updates more slowly than the frame rate (a 1 Hz battery feed at 60 fps is 59 wasted draws).

```tsx
<TimeSeries fps={30} mockMode height={180} />
```

15 fps is usable for very slow signals; below that the chart starts to feel laggy.

## Canvas DPI

Canvas defaults to 1 device pixel = 1 logical pixel — on retina displays that means everything is rendered at half the screen's native resolution and looks blurry. Altara's canvas components multiply the backing store by `window.devicePixelRatio` and scale the context to match. Nothing for you to configure.

## Memory profiling

To verify there's no leak:

1. Open Chrome DevTools → Performance tab.
2. Click ⏺ Record. Let your app run for 5–10 minutes with live data.
3. Stop. Look at the **JS Heap** track:
   - Healthy: sawtooth pattern (allocations + periodic GC) with no upward drift.
   - Leak: monotonic growth over time.

If you see growth, take heap snapshots at the start and end (DevTools → Memory tab → "Heap snapshot"), then **Comparison** view between them — the dominant retained objects are usually array buffers from data sources that weren't `destroy()`'d.

## Common pitfalls

- **`setState` in a high-frequency callback.** Subscribe to a data source via `useTelemetry` (which writes to a buffer) — don't call `setState` per sample.
- **Unbounded arrays.** If you build your own `AltaraDataSource`, cap its history with a fixed-size buffer or splice.
- **Forgetting `destroy()`.** Mock and adapter sources hold timers and sockets — call `destroy()` in your cleanup so they don't outlive the component.
- **`bufferSize` smaller than the visible window.** Samples drop out of the buffer before the chart's done with them — you see a clipped trace.
